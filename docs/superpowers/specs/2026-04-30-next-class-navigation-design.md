# 次の教室案内システム 設計書

## 1. 解決したい課題

学生は、次にどの教室へ行けばよいかを把握しづらく、毎回時間割表を見て確認する必要がある。
特に授業終了直後や空きコマ中に、次の移動先をすぐ判断できないことが不便である。

本システムは、学生が自分の履修授業を選択して個人用カレンダーを作成し、そのデータをもとに「次に行くべき教室」をすぐ確認できるようにする。

## 2. ver0.1 の目的

- 学生が学校の Google Workspace アカウントでログインできる
- 学生が授業一覧から自分の履修授業を選択できる
- 選択結果から自分用カレンダーを作成できる
- ホーム画面で次の授業の教室を最優先で確認できる
- 現在時刻に応じて表示内容が自動で切り替わる

## 3. ver0.1 のスコープ外

以下は ver0.1 では対応しない。

- CSV 取り込み
- 学校の既存システム連携
- 通知機能
- 休講、補講、教室変更への対応
- 学籍番号、所属学科、クラス情報の追加取得

## 4. 利用者

### 4.1 学生

- 学校の Google Workspace アカウントでログインする
- 授業一覧から自分の履修授業を選ぶ
- 自分用カレンダーを確認する
- ホーム画面で次に行く教室を確認する

### 4.2 管理者

- 学校の Google Workspace アカウントでログインする
- 事前に許可されたメールアドレスである場合のみ管理者として扱われる
- 授業と授業コマを登録、編集する

## 5. 認証と権限制御

### 5.1 認証方式

- 学生、管理者ともに Google ログインを使用する
- 学校の Google Workspace ドメインのアカウントのみ許可する
- 学校ドメイン以外の Google アカウントは拒否する

### 5.2 学生の初回ログイン

- 初回ログイン時に学生ユーザーを自動登録する
- Google アカウントの表示名を利用する
- 初回登録時に追加プロフィール入力は求めない

### 5.3 管理者判定

- あらかじめ許可したメールアドレスのみ管理者権限を持つ
- 学校ドメインのユーザーでも、許可リストにない場合は学生としてのみ利用できる

## 6. データ構成

### 6.1 授業

授業は学生が履修単位で選択する親データとする。
各授業は次の情報を持つ。

- 授業 ID
- 授業名

### 6.2 授業コマ

授業コマは、授業ごとの実際の開催枠を表す子データとする。
1つの授業は複数の授業コマを持てる。

各授業コマは次の情報を持つ。

- 授業コマ ID
- 授業 ID
- 曜日
- 時限
- 開始時刻
- 終了時刻
- 教室

この構成により、以下を表現できる。

- 月曜 1-2 限の連続授業
- 月曜 1 限と水曜 2 限のような週 2 コマ授業
- 金曜 3 限だけの週 1 コマ授業

### 6.3 履修データ

学生が選択した履修授業を次の対応で保存する。

- ユーザー ID
- 授業 ID

学生は授業を選択し、授業コマ単位では選択しない。

### 6.4 個人用カレンダー

- 個人用カレンダーは授業、授業コマ、履修データから生成する
- 学生ごとの週間予定として扱う
- 表示上は授業コマを時間枠として並べる

## 7. 主要画面

### 7.1 ホーム画面

最重要画面とする。

画面上部に大きく表示する情報:

- 次の授業
- 教室
- 開始時刻

表示ルール:

- 授業中は現在の授業情報を基準に表示する
- 授業終了 10 分前からは次の授業を優先表示する
- 空きコマ中は「空き時間中。次は 3 限・○○教室」のように表示する
- その日の授業が終わっていれば「本日の授業は終了」と表示する

### 7.2 履修登録画面

- 授業一覧を表示する
- 学生が自分の履修授業を選択できる
- 選択結果を履修データとして保存する
- 1つの授業に複数の授業コマがある場合は、それらを含めて1つの授業として選択する

### 7.3 自分用カレンダー画面

- 学生が登録した履修授業を確認できる
- 曜日ごとの予定を見返せる

### 7.4 管理者向け授業管理画面

- 管理者が授業と授業コマを登録、編集できる
- ver0.1 では授業名を授業データに持ち、曜日、時限、開始時刻、終了時刻、教室を授業コマに持たせる

## 8. 主な利用フロー

### 8.1 学生ログイン

1. 学生がアプリを開く
2. Google ログインを行う
3. システムが学校ドメインかを確認する
4. 初回ログインならユーザーを自動作成する
5. 学生画面へ遷移する

### 8.2 履修登録

1. 学生が履修登録画面を開く
2. 授業一覧から履修授業を選ぶ
3. システムが履修データを保存する
4. システムが選択した授業に紐づく授業コマをもとに個人用カレンダーを生成する

### 8.3 日常利用

1. 学生がホーム画面を開く
2. システムが曜日と現在時刻を確認する
3. 当日の授業コマを抽出する
4. 表示ルールに従って表示内容を決定する
5. 学生が次に行く教室を確認する

## 9. 表示判定ロジック

ログイン中の学生について、履修している授業に紐づく当日の授業コマを抽出し、開始時刻順に並べて判定する。

判定ルール:

- 履修授業が 1 件も未登録なら「履修授業を登録してください」と表示し、履修登録画面へ誘導する
- 現在が授業コマ中で、終了まで 10 分を超えている場合は現在の授業コマを基準に表示する
- 現在が授業コマ中で、終了まで 10 分以内の場合は次の授業コマを優先表示する
- 授業コマと授業コマの間なら「空き時間中。次は ○ 限・○○教室」と表示する
- その日の残り授業コマがなければ「本日の授業は終了」と表示する

## 10. エラー処理と例外ケース

- 履修未登録の場合は、エラー扱いで止めるのではなく、履修登録へ誘導する
- 授業コマに教室が未設定のデータがある場合は、授業名と開始時刻は表示しつつ、教室欄に「教室未設定」と明示する
- データ不備は隠さず表示する
- ver0.1 では休講、補講、教室変更は判定対象に含めない

## 11. 設計方針

システムは次の責務ごとに分けて設計する。

- 認証とユーザー管理
- 管理者向け授業、授業コマ管理
- 学生向け履修登録
- 授業、授業コマ、履修データからの個人用カレンダー生成
- ホーム画面の案内表示ロジック

特に、表示判定ロジックはデータ保存処理から分ける。
これにより、将来「授業終了 10 分前に通知する」機能を追加する際にも、同じ判定ルールを再利用しやすくする。

また、ver0.1 は最初から本番クラウド環境に載せるのではなく、まずローカルで機能を成立させ、その後 Cloudflare に段階的に移行できる構成にする。

### 11.1 開発初期の実行環境

- 開発初期はローカルの Node.js 環境で動かす
- 画面、認証、業務ロジックを先に成立させることを優先する
- 初期の開発用データベースは SQLite を使用する

### 11.2 永続化の分離

- 授業、授業コマ、履修データ、管理者判定に使う永続化処理は、画面や表示判定ロジックから分離する
- アプリ本体は、特定のデータベース実装に直接依存しない
- データアクセスは repository や service の境界を通して扱い、SQLite 実装から別実装へ差し替えやすくする

この方針により、開発初期は SQLite で素早く作りつつ、後から Cloudflare 用の D1 実装へ移しやすくする。

### 11.3 Cloudflare への移行前提

- 将来の配備先は Cloudflare Workers を前提とする
- Next.js は Cloudflare Workers 互換の実行方式で配備できる構成に寄せる
- 本番用データベースは Cloudflare D1 を利用する想定とする
- そのため、SQLite 固有の前提をアプリ全体へ広げない

### 11.4 環境分離

- Cloudflare 移行後は `staging` と `production` を分けて運用する
- `staging` と `production` は、それぞれ別の Worker、別の D1、別の環境変数を持つ
- Google ログインの設定も環境ごとに分離し、それぞれのリダイレクト先を管理する
- 管理者許可リストも環境単位で管理できるようにする

## 12. リリース段階

### 12.1 フェーズ 1: ローカル開発

- ローカル環境で学生向け機能と管理者向け機能を一通り動かす
- ホーム画面の表示判定が仕様通りに動くことを優先して固める

### 12.2 フェーズ 2: staging 環境

- 主要機能がローカルで安定した後、Cloudflare 上に `staging` を作る
- `staging` では Cloudflare Workers、D1、Google ログイン設定を接続し、実運用に近い状態で確認する

### 12.3 フェーズ 3: production 環境

- `staging` で問題がないことを確認した後、`production` を作る
- `production` は `staging` と分離した本番用の Worker、D1、環境変数を持つ

## 13. 完成条件

ver0.1 の機能完成は、まずローカル環境で次の条件を満たしたときとする。

1. 管理者が授業と授業コマを登録できる
2. 学生が学校ドメインの Google アカウントでログインできる
3. 学生が授業一覧から履修授業を選択できる
4. 学生が自分用カレンダーを確認できる
5. ホーム画面で時刻に応じて正しく表示内容が切り替わる

その後の配備完了条件として、`staging` と `production` に同等機能を展開できることを目指す。

## 14. テスト観点

最低限、次の観点を確認する。

- 通常の授業中に現在の授業が正しく表示されること
- 授業終了 10 分前から次の授業が優先表示されること
- 空きコマ中に次の授業と教室が表示されること
- その日の最後の授業後に「本日の授業は終了」と表示されること
- 履修未登録時に履修登録導線が表示されること
- 教室未設定の授業で「教室未設定」と表示されること
- SQLite 実装から D1 実装へ切り替えても、画面と判定ロジックの要件が変わらないこと

## 15. 将来拡張

将来的には次の拡張を想定する。

- 次の授業開始前の通知機能
- 休講、補講への対応
- 教室変更の反映
- 外部データ連携

ただし、これらは ver0.1 の対象外とし、まずは「自分の履修授業から次の教室がすぐ分かる」体験の実現を優先する。
